# SBI and Linux Sashiko

## SBI

---
RISC-V SBI (Supervisor Binary Interface) 完整分析                                                                                                                                  
                                                                                                                                                                                     
  概述                                                                                                                                                                               
                       
  RISC-V SBI 是运行在 M-mode（机器态）的固件向 S-mode（监管者态）操作系统提供的接口。Guest OS 中的 ecall 指令触发从 VS-mode → HS-mode（KVM）→ M-mode（SBI 固件）的陷阱，但 KVM
  会拦截并自行处理绝大多数 SBI 调用，只有特定调用会穿透到用户态（VMM）或转发到 M-mode 固件。

  ---
  SBI 扩展完整清单

  11. STA Extension (Steal Time Accounting) — ext_id: 0x535441 ("STA")

  让 Guest OS 感知自己被 Hypervisor 偷走了多少 CPU 时间（类似 x86 的 steal time）。

  ┌─────────────┬──────────────────────┬────────────────────────────────┐
  │ Function ID │         名称         │              作用              │
  ├─────────────┼──────────────────────┼────────────────────────────────┤
  │ 0           │ STEAL_TIME_SET_SHMEM │ 设置 steal time 的共享内存区域 │
  └─────────────┴──────────────────────┴────────────────────────────────┘

  Guest 在共享内存中写入 sequence（奇/偶校验序列号）和 steal（累计被偷时间）。KVM 在每个调度周期主动更新这些值。

  12. NACL Extension (Nested Acceleration) — ext_id: 0x4E41434C ("NACL")

  嵌套虚拟化加速接口。用于加速 L1 Hypervisor（如 KVM）管理 L2 Guest 的 CSR、HFENCE 和 SRET 操作。

  ┌─────────────┬───────────────┬────────────────────────────┐
  │ Function ID │     名称      │            作用            │
  ├─────────────┼───────────────┼────────────────────────────┤
  │ 0           │ PROBE_FEATURE │ 探测支持的 NACL 特性       │
  ├─────────────┼───────────────┼────────────────────────────┤
  │ 1           │ SET_SHMEM     │ 设置 NACL 共享内存         │
  ├─────────────┼───────────────┼────────────────────────────┤
  │ 2           │ SYNC_CSR      │ 同步 CSR 到共享内存        │
  ├─────────────┼───────────────┼────────────────────────────┤
  │ 3           │ SYNC_HFENCE   │ 同步 HFENCE 请求到共享内存 │
  ├─────────────┼───────────────┼────────────────────────────┤
  │ 4           │ SYNC_SRET     │ 同步 SRET 状态到共享内存   │
  └─────────────┴───────────────┴────────────────────────────┘

  NACL 特性：
  - SYNC_CSR (0x0)：通过共享内存批量同步 CSR
  - SYNC_HFENCE (0x1)：通过共享内存批量提交 HFENCE
  - SYNC_SRET (0x2)：通过共享内存优化 SRET 路径
  - AUTOSWAP_CSR (0x3)：硬件自动交换 CSR

  KVM 实现情况

  在 KVM 中已实现的扩展

  ┌───────────┬───────────┬───────────────────┬────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │   扩展    │  ext_id   │   KVM 实现方式    │                                                                  备注                                                                  │
  ├───────────┼───────────┼───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Base      │ 0x10      │ 本地处理          │ 在 vcpu_sbi_base.c 的 kvm_sbi_ext_base_handler 中直接返回 spec version、KVM 实现 ID (3)、Linux 版本码、vendor/march/mimpid 等          │
  ├───────────┼───────────┼───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ TIME      │ "TIME"    │ 本地处理          │ 在 vcpu_sbi_replace.c 中将调用转译为 kvm_riscv_vcpu_timer_next_event()，设置 vCPU 的虚拟定时器                                         │
  ├───────────┼───────────┼───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ IPI       │ "sPI"     │ 本地处理          │ 遍历 vCPU 列表通过 kvm_riscv_vcpu_set_interrupt(tmp, IRQ_VS_SOFT) 注入虚拟 IPI                                                         │
  ├───────────┼───────────┼───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ RFENCE    │ "RFNC"    │ 本地处理          │ 调用 kvm_riscv_fence_i() / kvm_riscv_hfence_vvma_*() 执行 TLB 刷新。HFENCE.GVMA/ VVMA 系列暂不支持（嵌套虚拟化未实现时返回             │
  │           │           │                   │ NOT_SUPPORTED）                                                                                                                        │
  ├───────────┼───────────┼───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤

  ├───────────┼───────────┼───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ HSM       │ "HSM"     │ 全本地处理        │ HART_START → 唤醒目标 vCPU；HART_STOP → 停止自身；HART_GET_STATUS → 返回状态；HART_SUSPEND ret_default →                               │

  ├───────────┼───────────┼───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ PMU       │ "PMU"     │ 本地处理          │ 通过 kvm_riscv_vcpu_pmu_* 系列函数管理虚拟 PMU 计数器。需要 CONFIG_RISCV_PMU_SBI                                                       │

  │ STA       │ "STA"     │ 本地处理          │ 在 vcpu_sbi_sta.c 中管理 guest 设置的共享内存，KVM 在每个调度周期通过 kvm_riscv_vcpu_record_steal_time() 更新 steal time               │
  ├───────────┼───────────┼───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ FWFT      │ "FWFT"    │ 本地处理          │ 在 vcpu_sbi_fwft.c 中管理特性配置，支持 MISALIGNED_EXC_DELEG 和 POINTER_MASKING_PMLEN（仅 64位），通过 vCPU 的 cfg 结构体和 CSR        │
  │           │           │                   │ 操作实现                                                                                                                               │

  │ V01       │           │                   │                                                                                                                                        │

  ├──────────────┼────────────┼───────────────────────────────────────────────────┼────────────────────┤
  │ SUSP         │ "SUSP"     │ 默认禁用，验证条件后转发到用户态 VMM 执行实际挂起 │ KVM_EXIT_RISCV_SBI │

  ├──────────────┼────────────┼───────────────────────────────────────────────────┼────────────────────┤

  关于 KVM_EXIT_RISCV_SBI：这是 KVM 的一种特殊退出类型，KVM 将 ecall 的全部参数（extension_id、function_id、a0–a5）填入 run->riscv_sbi 结构体后退出到用户态。用户态 VMM (如 QEMU/KVMTool)
  处理后通过 KVM_RISCV_SBI_SET_RETURN 设置返回值，然后 KVM_RISCV_SBI_RETURN_DONE 让 KVM 恢复 guest。

  ---
  Guest 执行 ecall 触发各 SBI 时发生了什么

  总体流程

  Guest OS (VS-mode)
      │ ecall 指令
      ▼
  HS-mode 陷阱 (KVM.8)
      │
      ▼
  kvm_riscv_vcpu_sbi_ecall()   ← vcpu_sbi.c
      │
      ├── kvm_vcpu_sbi_find_ext(vcpu, a7=ext_id)
      │   └── 在 sbi_ext[] 表中查找匹配扩展
      │
      ├── ext->handler(vcpu, run, &retdata)
      │   └── 按扩展类型分发:
      │       ├── 本地处理 → 设置 a0/a1 返回值
      │       ├── 转发到用户态 → 设置 KVM_EXIT_RISCV_SBI
      │       └── trap 重定向 → 注入异常到 guest
      │
      └── sepc += 4  (跳过 ecall 指令)

  各扩展的具体行为

  BASE (0x10)

  - PROBE_EXTENSION(a0=ext_id): KVM 在 sbi_ext[] 中查找 ext_id。如果是实验性/厂商扩展范围 → 转发到用户态。否则返回是否存在。
  - 其他查询: 直接从 KVM 维护的数据结构返回（spec_version=KVM_SBI_VERSION_MAJOR.KVM_SBI_VERSION_MINOR=3.0，impl_id=3=KVM，impl_version=LINUX_VERSION_CODE，mvendorid/marchid/mimpid 来自
  vCPU 架构配置）
  - Guest 视角: 函数调用直接返回，无 VM exit

  TIME ("TIME")

  - Guest 设置 a6=0 (SET_TIMER), a0=next_cycle
  - KVM 调用 kvm_riscv_vcpu_timer_next_event(vcpu, next_cycle) 设置虚拟 htimedelta 或 VS-level 定时器
  - Guest 视角: 定时器到期时收到 VS-level timer interrupt

  IPI ("sPI")

  - Guest 设置 a0=hmask (hart mask), a1=hbase (hart base)
  - KVM 遍历所有 vCPU，对匹配目标通过 kvm_riscv_vcpu_set_interrupt(tmp, IRQ_VS_SOFT) 注入软件中断
  - Guest 视角: 目标 hart 收到 VS-mode 软件中断，调用 IPI handler

  RFENCE ("RFNC")

  - Guest 设置 a0=hmask, a1=hbase, a6=function_id, a2-a4=地址范围
  - KVM 执行 kvm_riscv_fence_i() 或 kvm_riscv_hfence_vvma_*()，这些函数最终操作 G-stage 页表或向物理 CPU 发送 fence
  - HFENCE.GVMA/VVMA 系列: 由于 KVM 不支持嵌套虚拟化，一律返回 NOT_SUPPORTED
  - Guest 视角: 调用完成后 TLB 已刷新

  HSM ("HSM")

  - HART_START(a0=target_vcpuid, a1=start_pc, a2=priv): KVM 找到目标 vCPU，调用 kvm_riscv_vcpu_sbi_request_reset() 设置启动 PC 和 a1 参数，然后 __kvm_riscv_vcpu_power_on() 唤醒。使用
  cross_vcpu_lock 与 SUSP/SRST 序列化
  - HART_STOP: 调用 __kvm_riscv_vcpu_power_off() 将 vCPU mp_state 设为 STOPPED
  - HART_GET_STATUS(a0=target_vcpuid): 返回 STOPPED/STARTED/SUSPENDED
  - HART_SUSPEND(a0=suspend_type):
    - SUSPEND_RET_DEFAULT(0x0): 调用 kvm_riscv_vcpu_wfi() → 即 WFI 行为
    - SUSPEND_NON_RET_DEFAULT(0x80000000): 不支持 → 返回 NOT_SUPPORTED
  - Guest 视角: 启动其他 hart 时，新 hart 从指定 PC 开始执行（a0=hart_id, a1=priv）；停止或挂起时被调度走

  SRST ("SRST")

  - SYSTEM_RESET(a0=type, a1=reason):
    - SHUTDOWN: kvm_riscv_vcpu_sbi_system_reset() 停止所有 vCPU → KVM_EXIT_SYSTEM_EVENT type=SHUTDOWN → 用户态关机
    - COLD_REBOOT/WARM_REBOOT: 同样停止所有 vCPU → KVM_EXIT_SYSTEM_EVENT type=RESET → 用户态重启
  - Guest 视角: 调用后不会返回（如果用户态实现了对应操作）

  PMU ("PMU")

  - 完整的功能函数映射到 kvm_riscv_vcpu_pmu_* 系列函数
  - 使用 perf core 框架创建和管理事件
  - 支持硬件计数器和固件事件计数器
  - Guest 视角: 类似于访问硬件 PMU，可以开始/停止/读取性能计数器

  STA ("STA")

  - STEAL_TIME_SET_SHMEM(a0=shmem_phys, a1=shmem_phys_hi, a2=flags): Guest 提供一块共享内存的物理地址（需64字节对齐）
  - KVM 记录 vcpu->arch.sta.shmem = shmem
  - 在 vCPU 被调度时，KVM 的调度 hook 调用 kvm_riscv_vcpu_record_steal_time() 更新共享内存中的 steal 字段（累积被偷时间）和 sequence 字段（奇偶校验）
  - Guest 视角: 周期性地读取共享内存，计算被偷时间，用于调度器负载均衡

  SUSP ("SUSP") — 默认禁用

  - SYSTEM_SUSPEND(a0=sleep_type, a1=resume_addr, a2=resume_priv): 仅支持 SUSPEND_TO_RAM(0)
  - KVM 检查：当前是否在 S-mode？所有其他 vCPU 是否已 STOPPED？resume_addr 是否有效？
  - 验证通过后，将挂起信息记录到 vCPU reset_state，然后 KVM_EXIT_RISCV_SBI 转发到用户态
  - Guest 视角: 系统进入挂起状态，恢复时从 resume_addr 开始执行

  FWFT ("FWFT")

  - SET(a0=feature, a1=value, a2=flags): KVM 在 features[] 表中查找 feature，验证是否支持，加锁保护（可用 LOCK 标志防止后续修改），然后应用特性值
  - GET(a0=feature): 查询当前特性值
  - KVM 实现的特性：
    - MISALIGNED_EXC_DELEG: 操作 HEDELEG CSR 的 misaligned load/store 位
    - POINTER_MASKING_PMLEN (仅64位): 操作 HENVCFG CSR 的 PMM 位
  - Guest 视角: 可动态调整固件/硬件行为

  DBCN ("DBCN") — 默认禁用，转发到用户态

  - 所有 console 读写请求通过 KVM_EXIT_RISCV_SBI 退出到用户态 VMM
  - VMM 执行实际的字符设备 I/O
  - Guest 视角: 通过 SBI 调用的控制台 I/O

  MPXY ("MPXY") — 默认禁用，转发到用户态

  - Guest 通过标准 MPXY 接口（GET_SHMEM_SIZE、SET_SHMEM、SEND_MSG 等）发送消息
  - KVM 全部转发到用户态 VMM，VMM 实现底层的 RPMI 或其他消息协议
  - Guest 视角: 可以通过标准 SBI 接口与板载固件（如 BMC、电源管理）通信

  Experimental / Vendor — 转发到用户态

  - KVM 不做任何处理，直接 KVM_EXIT_RISCV_SBI 交给 VMM

  1. SBI 共定义了 15 个标准扩展（含 Legacy），加 Experimental/Vendor 范围                                                                                                            
  2. KVM 本地实现了 8 个扩展（Base, TIME, IPI, RFENCE, SRST, HSM, PMU, STA, FWFT）— 这些调用在 KVM 内部完成，guest 无 VM exit                                                        
  3. KVM 转发到用户态的 4 个扩展（SUSP, DBCN, MPXY）— 默认禁用，启用后通过 KVM_EXIT_RISCV_SBI 退出到 VMM
  4. KVM 未实现的 2 个扩展（DBTR 完全未实现；NACL 只在 KVM 内部使用，不暴露给 guest）                                                                                                
  5. 所有转发型的扩展（Experimental、Vendor、DBCN、MPXY、SUSP）都使用 kvm_riscv_vcpu_sbi_forward_handler 统一退出到用户态

## Sashiko

Sashiko 是什么？                                                                                                                                                                     
  
  Sashiko（刺し子，日语意为"小针脚"）是一个面向 Linux 内核的智能化代码审查系统。名称的隐喻来自于日本传统的刺绣工艺——用细密的针脚加固布料——这里指用自动化的 AI 审查来加固 Linux         
  内核的补丁质量。                    
                    
  它是 Linux Foundation 旗下的开源项目，完全自包含（不依赖任何外部 agent 工具），用 Rust（Axum 框架）构建，支持多种 LLM 提供商（Gemini、Claude、OpenAI、Bedrock、Vertex 等）。

  据其基准测试，Sashiko 能够发现 53.6%（使用 Gemini 3.1 Pro）的上游合入后才发现 bug 的补丁——某种意义上已经超过了人类 reviewer（因为这些 bug 100% 通过了人工审查）。

  关键设计点：

  - 三阶段管道：raw_tx(原始事件) → parsed_tx(解析后文章) → DB Worker(批量入库)
  - 通道缓冲：两个 mpsc::channel(1000) 实现异步解耦，Parser Dispatcher 用 Semaphore(50) 控制并发解析
  - 摄入截止时间：通过 cutoff_timestamp 避免重复摄入旧邮件
  - 启动恢复：reset_reviewing_status() 在重启时把之前中断的 InReview 状态重置为 Pending

  2. 审查调度器 (reviewer.rs)

  这是最核心的部分。Reviewer 是一个后台循环服务：

  // 伪代码示意
  loop {
      // 1. 从 DB 取出 Pending 的 patchset
      let patchsets = db.get_pending_patchsets(10).await;

      // 2. 对每个 patchset，spawn 异步任务
      for patchset in patchsets {
          tokio::spawn(review_patchset_task(context, patchset));
      }

      // 3. 检查是否有已到期的 embargo（延迟发布）
      release_embargoed_results().await;

      // 4. 每 10 秒轮询一次
      sleep(10s);
  }

  review_patchset_task 是每个 patchset 审查的完整生命周期：

  1. 状态更新：Pending → InReview
  2. 获取 diff：从 DB 读取所有 patch 的 diff
  3. Baseline 解析：通过 BaselineRegistry 尝试找到合适的 baseline（可应用补丁的基线提交）
  4. 创建 worktree：在 baseline 上 git checkout 并创建 git worktree
  5. 应用补丁：尝试 git am → 回退 git apply → 或直接 checkout commit
  6. 运行 review binary：spawn review 子进程，传入 JSON payload
  7. 处理结果：解析 findings、记录 token 用量、调用 queue_notifications
  8. 清理：删除 worktree

  ★ 并发控制：对于包含 ≥10 个 patch 的大型 patchset，会自动 try_acquire 额外的 semaphore permit 来并行审查多个 patch，形成"主 worker + 辅助 worker"的模式。

  3. 10 阶段审查协议（核心创新）

  这是 Sashiko 最有特色的设计——不是让 LLM 一次性审查整个补丁，而是模拟一个由不同专长的 reviewer 组成的团队，分阶段审查：

  Phase 0: Pre-screening
    │  └── AI 选择相关的子系统指南（如 mm.md、locking.md）
    │
  Planning Phase
    │  └── AI 决定 Stages 4-7 中哪些需要运行
    │
    ├── Stage 1: 分析 commit 主要目标（架构缺陷、UAPI 破坏）
    ├── Stage 2: 高级实现验证（commit message 是否匹配代码？）
    ├── Stage 3: 执行流验证（逻辑错误、off-by-one、NULL 解引用）
    ├── Stage 4: 资源管理（内存泄漏、UAF、double free）
    ├── Stage 5: 锁与同步（死锁、RCU 规则、原子上下文睡眠）
    ├── Stage 6: 安全审计（缓冲区溢出、越界、TOCTOU）
    ├── Stage 7: 硬件工程师审查（寄存器访问、DMA、memory barrier）
    │
    ├── Stage 8: 去重与合并（合并 Stages 1-7 的重叠问题）
    ├── Stage 9: 验证与严重性评估（过滤假阳性，分配 severity）
    └── Stage 10: LKML 报告生成（生成格式化的内联审查回复）

  Stage 3-6 还有一个特殊的优化：不把 diff 内容发到 AI 的 system prompt 里（用 shared_context_no_log），因为 AI 可以自己通过工具读取源码。这能节省大量上下文 token。

  每个 Stage 都有给 AI 的专用角色扮演提示词，例如 Stage 5 让 AI 扮演"世界级的并发与锁专家"，Stage 6 扮演"红队安全研究员"。

  4. AI 提供者抽象层 (ai/mod.rs)

  5. Stdio 协议（review binary 通信）

  Daemon 和 review binary 之间的通信协议是一个基于 stdin/stdout 的 JSON 行协议：

  Daemon                              Review Binary
    │                                       │
    │── stdin: JSON payload (patchset) ────▶│
    │                                       │── 工具调用（git、文件读取等）
    │◀── stdout: {"type":"ai_request",      │
    │              "payload": AiRequest}────│
    │                                       │
    │── stdin: {"type":"ai_response",       │
    │            "payload": AiResponse}────▶│
    │                                       │
    │         ... 多轮对话 ...               │
    │                                       │
    │◀── stdout: {"patchset_id":...,        │
    │              "review":...,             │
    │              "findings":[...]}─────────│

  这个设计实现了职责分离：daemon 只负责任务调度、DB 操作和邮件发送；实际的 AI 对话循环在 review binary 中运行。Daemon 作为一个"AI 代理中间人"，拦截 ai_request 消息并调用实际的 AI
  provider，然后返回 ai_response。

  6. Worker 与 ToolBox（工具使用）

  Worker 是 review binary 中的 AI 会话管理器：

  pub struct Worker {
      provider: Arc<dyn AiProvider>,
      tools: ToolBox,
      prompts: PromptRegistry,
      global_history: Vec<AiMessage>,
      max_interactions: usize,  // 每阶段最大交互轮数
      temperature: f32,
      stages: Option<Vec<u8>>,
  }

  AI 可用的工具有 13 个，封装在 ToolBox 中：

  - 代码阅读：read_files（支持 smart 模式，自动折叠无关代码）、search_file_content（基于 grep）
  - Git 操作：git_blame、git_diff、git_show、git_log、git_status、git_checkout、git_branch、git_tag
  - 文件操作：list_dir、find_files
  - 辅助：TodoWrite、read_prompt

  ★ 安全设计：validate_path() 对路径遍历攻击做了严格防护——通过 canonicalize() 解析符号链接后再检查路径是否在 worktree 范围内。

  ★ Git 参数白名单：validate_git_args() 对 git 命令参数做白名单校验，防止 --output=malicious.txt 等危险参数。

  7. 预取上下文 (worker/prefetch.rs)

  在 AI 审查开始前，Sashiko 会基于 diff 中修改的行，自动用 tree-sitter 解析 C 代码的 AST，提取被修改的函数、结构体的完整源代码，作为预取上下文发给 AI。这样 AI
  不需要通过工具读取文件就能看到相关的完整上下文——显著减少交互轮数。

  8. 通知与邮件系统

  审查完成后，Reviewer 调用 queue_notifications()：

  1. Patchwork 集成：将审查状态（warning/success）通过 API 发布到 Patchwork 实例
  2. Email outbox：将待发送的邮件插入 DB 的 email_outbox 表
  3. EmailRouter：根据 email_policy.toml 决定邮件发送策略——发往哪些列表、是否包含积极反馈等
  4. Embargo 机制：某些子系统（如 netdev）的审查结果可以延迟 24 小时发布，给作者时间先看到结果

  9. Baseline 解析 (baseline.rs)

  BaselineRegistry 是识别补丁应该基于哪个内核版本审查的核心：

  - 解析内核 MAINTAINERS 文件中每个子系统的 T: 树条目
  - 在补丁涉及的子系统中查找对应的 git 树 URL
  - 按优先级尝试 local ref → commit hash → remote fetch
  - 支持自定义 remote（Settings.git.custom_remotes）

  ★ Insight ─────────────────────────────────────
  Sashiko 的设计哲学非常有意思：它不是让一个单一的 LLM "看一眼"补丁然后给出意见，而是通过 10 阶段的序列化审查协议 + 角色扮演 prompt（每个 stage 扮演不同的专家角色）+ 工具使用（让 AI
  能主动读取源码、git blame、搜索） 来系统性地提升审查质量。每个 stage 的 prompt 都针对 Linux 内核的开发特点精心设计，比如 Stage 5 明确列出了 9
  类并发问题检查项。这种"多视角交叉验证"的思路是 Sashiko 能超过人类 reviewer（在某些指标上）的关键原因。
